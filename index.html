Frequently Asked Questions

What is Tightwad?

Tightwad pools your mixed CUDA, ROCm, and Metal GPUs into a single OpenAI-compatible API endpoint. It uses speculative decoding to accelerate inference: a small draft model generates candidate tokens quickly, and a large target model verifies them in batch. Output quality is identical to running the target model alone.

Does speculative decoding change the output quality?

No. With greedy decoding (temperature=0), the output is mathematically identical to running the target model alone. Speculative decoding is a pure speed optimization — it finds the same tokens faster by batch-verifying draft predictions.

What hardware do I need?

At minimum: two machines (or two inference servers on one machine). One runs a small draft model (1.7B-8B, runs on any CPU or cheap GPU). The other runs your target model (32B-70B+). They communicate over HTTP — LAN, WiFi, or even across the internet.

Do the draft and target models need to be the same family?

Yes. This is the #1 requirement. Llama draft → Llama target (good). Qwen draft → Llama target (bad — 1.6% acceptance, 10x slower). Tightwad auto-detects model families at startup and warns on mismatch.

What's the difference between max_draft_tokens: 32 and auto?

auto adjusts at runtime based on your acceptance rate. High acceptance (>80%) → drafts more tokens per round. Low acceptance (<40%) → drafts fewer. It also factors in draft vs verify timing — if your draft model is fast and your target is slow, it drafts more aggressively. Most users should use auto.

Can I use Ollama as a backend?

Yes. Set backend: ollama for any endpoint. Ollama works for both draft and target. However, llama-server backends give better performance because they support logprobs-based verification (batch speedup), while Ollama falls back to text-match comparison.

What is the quality gate mode?

Quality gate (tightwad gate start) is a different mode from speculation. A fleet of cheap CPU/GPU agents generate full responses, and a powerful GPU reviews each one — approving, correcting, or rejecting it. 60-80% of responses pass unchanged, so the GPU only processes the hard cases. Great for datacenter cost reduction.

What is consensus verification?

When you have multiple drafters, consensus mode compares their outputs before asking the target. If all drafters agree on a token, it's accepted immediately — no target call needed. Three modes: strict (unanimous), majority (>50%), any_disagree (verify on disagreement).

How does the peer agent work?

The peer agent (tightwad peer start) is a lightweight HTTP daemon on port 9191. It replaces SSH for managing remote nodes — version checks, GPU info, model inventory, starting/stopping rpc-server. Works on Windows, macOS, and Linux. Configure peer_port on workers to use it.

Does it work over WiFi?

Yes. Our benchmark (1.86x speedup on 70B) was measured over WiFi. RPC pool mode is slower over WiFi (~2.2 tok/s) but speculation-over-RPC amortizes the latency (4.1 tok/s). For pure speculation proxy mode (no RPC), WiFi is fine — only token IDs (bytes) are sent, not tensor data.

Best Practices

Always use same-family models. Draft and target must share the same architecture (Llama → Llama, Qwen → Qwen). Cross-family pairs get <5% acceptance. Tightwad detects this automatically.
Use max_draft_tokens: auto. The cost-aware auto-tuner adjusts based on your actual draft and verify speeds. No need to guess.
Use llamacpp backend when possible. It supports logprobs-based batch verification, which is the core speedup mechanism. Ollama falls back to text-match comparison (still works, but slower).
Run tightwad doctor before starting. It checks config, binaries, network, versions, model families, and MoE compatibility in one command.
Pin llama.cpp versions. All nodes must run the same llama.cpp build. tightwad start enforces this by default. Mismatched versions cause silent failures.
MoE models have hidden VRAM overhead. Routing tables are replicated to every GPU. A 12GB GPU can't handle a 128-expert MoE model even if tensor split only assigns it 14% of weights. tightwad doctor warns about this.
Use the peer agent on Windows. Windows machines often lack SSH. The peer agent provides the same management capabilities via HTTP. Set peer_port: 9191 on worker entries.
Benchmark before and after. Run tightwad bench to get a concrete speedup number. It runs the same prompts through the proxy and directly against the target, showing per-prompt tok/s comparison.

Real-World Use Cases

Homelab GPU Pooling

Pool a desktop RTX 4070 Ti + an old RTX 2070 + a Mac Mini M2 into one endpoint. Run 70B models that don't fit on any single machine. Speculation-over-RPC gives 1.86x speedup.

Cloud Cost Reduction

Draft locally on a cheap GPU, verify via cloud API. 80% of tokens come from your local hardware. 5x reduction in API output token costs.

Datacenter CPU Fleet

Use quality gate mode: 100 idle CPUs run 1.7B models, one H100 reviews outputs. 60-80% pass unchanged. 100x throughput at 20-40% GPU cost.

Multi-Drafter Consensus

Three cheap machines each draft tokens. When they agree, skip the expensive target entirely. Best for scenarios where accuracy matters more than speed.

Mixed Vendor Cluster

NVIDIA CUDA + AMD ROCm + Apple Metal in the same cluster. Tightwad doesn't care about GPU vendor — it pools them all into one API.

Legacy Hardware Revival

That dusty GTX 770 with 2GB VRAM? It can run a 1.7B draft model at 30 tok/s. Pair it with a modern GPU running 70B. Every GPU contributes.

Troubleshooting

Low acceptance rate (<10%)

Almost always a model family mismatch. Run tightwad doctor to check. The draft and target must be the same architecture family (e.g., both Llama, both Qwen). Also check that you're not mixing Llama 3.2 with Llama 3.3 — despite the name, Llama 3.2 is a different architecture from 3.1/3.3.

Chat completions work but acceptance is 0%

Likely a chat template mismatch. v0.4.0+ auto-detects templates, but if detection fails it defaults to ChatML (Qwen format). Set proxy.chat_template: llama3 explicitly for Llama models. Use /v1/completions with a pre-formatted prompt to bypass templates entirely.

alloc_tensor_range: failed to allocate

MoE model replication overhead. MoE models replicate routing tables to every GPU, adding ~20GB per device. A 12GB GPU can't handle this even with small tensor split share. Use only 24GB+ GPUs for MoE models, or run on a single machine with enough total VRAM.

Coordinator hangs at startup

Usually a llama.cpp version mismatch between coordinator and workers. tightwad start checks this automatically. If it can't SSH to workers, use --skip-version-check after verifying manually. Also check that all rpc-server instances are running.

Proxy slower than direct target

If the target is already fast (>50 tok/s), HTTP round-trip overhead can negate speculation gains. This is expected for small models on powerful GPUs. Speculation helps most when the target is the bottleneck (large models, pooled GPUs, slow hardware). Run tightwad bench to measure your actual speedup.

Datacenter Deployment

Architecture

At datacenter scale, Tightwad's quality gate mode turns idle CPUs into productive inference nodes:

CPU 1 (Qwen3-1.7B) ── response ──┐
CPU 2 (Qwen3-1.7B) ── response ──┤
CPU 3 (Qwen3-1.7B) ── response ──┼──> Quality Gate ──> H100 (405B) ──> verified output
...                               │     approve/correct/reject
CPU N (Qwen3-1.7B) ── response ──┘

Cost Model

MetricWithout TightwadWith Quality Gate

Requirements

  • CPU agents need enough RAM for a 1.7B-8B model (~2-8GB)
  • GPU verifier needs the full target model in VRAM
  • Network: standard LAN is fine (only text, not tensors)
  • Install peer agents on all nodes for management: tightwad peer start
